## Context

Voboost is delivered as two APKs:

- **`voboost`** (this repo, `ru.voboost`) — the client app. It bundles the UI,
  settings, translations, and the Frida agent scripts as carrier assets. A new
  release is a new APK installed over the previous one.
- **`voboost-inject`** (sibling repo) — the root daemon. A separate APK, updated
  less often. It is installed by `adb install`, or **self-updated**: voboost
  stages a new daemon APK into the app zone and signals it; the daemon verifies
  the staged APK and replaces itself, then init restarts it. That self-update
  mechanism is owned by the voboost-inject repo.

There is deliberately **no partial update of agents/resources**. Earlier
file-level incremental designs (per-file diff, content-addressed agent staging,
a daemon-manifest producer in the client) are dropped — every update is a
complete APK, so the manifest lists APKs, not individual files.

The daemon manifest (`manifest.json` + `manifest.sig`, per-agent metadata that
the daemon verifies against its embedded key) is produced **build-time** and
ships inside the daemon APK — not generated by the client at runtime.

## Goals / Non-Goals

**Goals:**

- A signed release manifest lists the current `voboost` (app) and
  `voboost-inject` (core) APK versions; the client verifies it before use.
- The client downloads a newer APK whole only when its version is behind.
- Every downloaded APK byte is verified (size pre-check + sha256).
- `app` APK → installer replaces the running app; `core` APK → staged + signalled
  for the daemon to self-update.
- Never persist a manifest that fails verification.

**Non-Goals:**

- Partial/incremental updates of agents, scripts, or any "resources" — APK only.
- Producing or signing the daemon `manifest.json` at runtime — build-time, in
  the daemon APK.
- The daemon's `core` self-update (verify + replace + restart) — owned by
  voboost-inject.
- A general-purpose package manager; scoped to the two voboost APKs.

## Decisions

### D1. APK-level, never file-level
The release manifest lists whole APKs (`path`, `channel ∈ {app, core}`, `sha256`,
`size`, `version`). The client never diffs or stages individual agent files.
This removes the "resources" problem class entirely.

### D2. Version-gated download
The client compares the installed app version (its own `versionName`) and the
installed daemon version (read from `inject-status.json`) against the manifest.
An APK is downloaded only when the manifest version is newer. No version info in
the manifest ⇒ no update for that channel.

### D3. Two apply paths, by channel
- **`app`** — the client stages the verified APK and invokes the installer
  (`android.content.Intent` `ACTION_INSTALL_PACKAGE` /
  `PackageInstaller`, or `voboost-install`) to replace the running app.
- **`core`** — the client stages the verified APK into the app zone and writes a
  single-use marker; voboost-inject observes the marker, re-verifies the APK
  against its embedded trust, and self-replaces. The client never installs the
  daemon.

### D4. Whole-file download, sha256 authoritative
`size` is a DoS guard (quick reject of a truncated/oversized download before
hashing); `sha256` is the integrity check. No binary deltas.

### D5. ed25519 verify in the app
The release manifest signature verifies against the committed
`release-public.pem` via BouncyCastle (API 28 lacks `java.security` EdDSA). The
daemon verifies its own material with its embedded key — separate trust anchors,
same key family.

### D6. Never persist a failed manifest
A manifest that fails signature or structural validation is never written as the
current manifest; the client keeps the last good one.

### D7. One marker per staged core APK
The `core`-apply marker is single-use: the daemon consumes (deletes) it after
any apply attempt, so a successful self-update does not loop. The `app`-apply
has no marker — it ends in an install intent.
